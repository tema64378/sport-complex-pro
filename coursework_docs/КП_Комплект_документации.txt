# Аннотация

Курсовой проект посвящен разработке локальной информационной системы управления спортивным комплексом.

В рамках работы рассмотрены теоретические основы проектирования информационных систем для предметной области фитнес- и спортивных услуг, выполнен анализ требований, обоснован выбор технологического стека и разработана архитектура программного комплекса.

Программная реализация выполнена в виде клиент-серверного веб-приложения. Клиентская часть реализована на React (Vite), серверная часть — на Rust (Axum), хранение данных организовано в SQLite. Система обеспечивает базовые функции управления пользователями, клиентами, расписанием, бронированиями, платежами и отчетностью.

Практическая часть включает описание этапов разработки, структуры проекта, модели данных, API-взаимодействия, сценариев тестирования и руководств по эксплуатации.

Результатом курсового проекта является работоспособная локальная система, комплект проектной документации и материалы для защиты.

Ключевые слова: информационная система, спортивный комплекс, Rust, React, SQLite, Axum, автоматизация, веб-приложение, UML, IDEF.

## 1 Введение

Цифровизация процессов в сфере спортивных услуг требует разработки удобных и надежных информационных систем, позволяющих автоматизировать учет клиентов, занятий, абонементов, платежей и управленческой отчетности. Ручные подходы и разрозненные таблицы не обеспечивают необходимого уровня актуальности данных и затрудняют оперативное управление.

Цель курсового проекта — разработка локальной информационной системы управления спортивным комплексом и формирование полного комплекта документации для ее внедрения и демонстрации.

Задачи проекта:

1. Выполнить анализ предметной области и сформировать требования.
2. Спроектировать архитектуру программного комплекса.
3. Реализовать клиентскую и серверную части приложения.
4. Разработать схему данных и обеспечить целостность хранения.
5. Провести тестирование и оформить результаты.

Объект исследования: процессы управления деятельностью спортивного комплекса.

Предмет исследования: методы проектирования и реализации локальной информационной системы для автоматизации процессов учета и обслуживания.

---

## 2 Общая часть

### 2.1 Характеристика предметной области

В рамках деятельности спортивного комплекса выделяются ключевые информационные потоки:

1. Регистрация и ведение карточек клиентов.
2. Формирование расписания занятий и назначение тренеров.
3. Оформление бронирований и контроль посещаемости.
4. Учет абонементов, услуг и финансовых операций.
5. Формирование аналитических отчетов и сводных данных.

Для данной предметной области критичны следующие свойства системы:

1. Надежность хранения и целостность данных.
2. Контроль прав доступа по ролям.
3. Предсказуемое поведение интерфейса и API.
4. Возможность расширения состава сущностей и отчетов.

### 2.2 Анализ и уточнение требований

Функциональные требования:

1. Авторизация и работа с ролями пользователей (администратор, тренер, клиент).
2. CRUD-операции по основным сущностям: пользователи, клиенты, тренеры, занятия, бронирования, платежи.
3. Просмотр и редактирование расписания.
4. Формирование отчетов и экспорт табличных данных.
5. Поддержка административных операций с данными.

Нефункциональные требования:

1. Локальный запуск без внешнего сервера и домена.
2. Быстрое время отклика интерфейса в типовых сценариях.
3. Минимальные требования к инфраструктуре.
4. Сопровождаемость и читаемая структура проекта.

### 2.3 Обоснование выбора технологий

Серверная часть реализована на Rust (фреймворк Axum), что обеспечивает:

1. Типобезопасность и строгую обработку ошибок.
2. Высокую производительность и стабильность работы.
3. Надежность за счет компиляционных гарантий.

Клиентская часть реализована на React + Vite:

1. Компонентный подход упрощает развитие интерфейса.
2. Vite ускоряет цикл разработки и сборки.
3. Удобная интеграция с REST API.

Хранилище данных — SQLite:

1. Не требует отдельного серверного процесса БД.
2. Поддерживает транзакционную модель ACID.
3. Подходит для локальных и учебных систем.

### 2.4 Архитектура программного комплекса

Приложение построено по клиент-серверной модели:

1. Frontend (`frontend/`) обеспечивает пользовательский интерфейс.
2. Backend (`backend/`) реализует API и бизнес-логику.
3. SQLite (`backend/data.db`) хранит данные предметной области.

Взаимодействие компонентов происходит по HTTP/JSON через маршруты `/api/*`.

## 3 Практическая часть

### 3.1 Проектирование структуры данных

Для хранения данных выделены основные таблицы:

1. `users` — учетные записи, роли, данные профиля.
2. `sessions` — сессии авторизации.
3. `members` — карточки клиентов.
4. `trainers` — сведения о тренерах.
5. `classes` и `calendar_slots` — занятия и расписание.
6. `bookings` — записи клиентов.
7. `payments`, `receipts` — платежные операции.
8. Дополнительные таблицы аналитики и служебных данных.

Проектирование выполнено с учетом целостности: первичные ключи, внешние связи и обязательные поля.

### 3.2 Реализация серверной части

Серверная часть реализована в `backend/src/main.rs` на Axum.

Ключевые реализованные механизмы:

1. Маршрутизация API для всех функциональных разделов системы.
2. Проверка прав доступа и аутентификация по токенам.
3. Валидация входных данных.
4. Обработка ошибок с корректными HTTP-статусами.
5. Работа с SQLite через подготовленные запросы.

### 3.3 Реализация клиентской части

Клиентская часть размещена в `frontend/` и реализована на React.

Реализованы следующие группы интерфейсов:

1. Авторизация и регистрация пользователей.
2. Главная панель и аналитические карточки.
3. Работа с клиентами, тренерами и расписанием.
4. Модули платежей, отчетов и административной панели.
5. Динамические таблицы, фильтры, поиск, формы ввода.

### 3.4 Интеграция модулей

Интеграция выполнена через единый API-клиент фронтенда (`frontend/apiClient.js`).

Принципы интеграции:

1. Централизация HTTP-запросов.
2. Унифицированная обработка ошибок API.
3. Согласованный формат данных между frontend и backend.
4. Локальный запуск через `start-dev.sh`.

### 3.5 Тестирование и результаты

Проведена проверка работоспособности основных сценариев:

1. Авторизация и управление сессией.
2. CRUD-операции по ключевым сущностям.
3. Отображение отчетных и аналитических данных.
4. Работа ролей и ограничений доступа.

Техническая проверка проекта:

1. `cargo check` для backend — успешно.
2. `npm run build` для frontend — успешно.

Результат: система функционирует в локальном режиме и готова к демонстрации.

# Руководство пользователя

## 1 Назначение системы
Система предназначена для работы сотрудников и клиентов спортивного комплекса в локальном режиме.

## 2 Запуск

1. Открыть терминал.
2. Выполнить команды:

```bash
cd /Users/akovlevartem/Documents/курсовая
bash start-dev.sh
```

3. Открыть `http://localhost:3000`.

## 3 Основные роли

1. Администратор — полный доступ.
2. Тренер — доступ к клиентам, расписанию, бронированиям.
3. Клиент — личный кабинет, записи, платежные сведения.

## 4 Основные сценарии

### 4.1 Вход в систему

1. Перейти на страницу авторизации.
2. Ввести логин и пароль.
3. Нажать кнопку входа.

### 4.2 Работа с клиентами

1. Открыть раздел «Клиенты».
2. Добавить или изменить карточку клиента.
3. Сохранить изменения.

### 4.3 Работа с расписанием

1. Открыть раздел «Календарь» или «Тренировки».
2. Выбрать дату и тренера.
3. Добавить или изменить слот занятия.

### 4.4 Бронирования

1. Открыть раздел «Бронирования».
2. Выбрать клиента и занятие.
3. Подтвердить запись.

### 4.5 Платежи

1. Открыть раздел «Платежи».
2. Создать запись о платеже.
3. Проверить статус операции.

## 5 Типовые ошибки и действия пользователя

1. «Неверный логин/пароль» — проверить раскладку, повторить ввод.
2. «API недоступно» — убедиться, что backend запущен на порту 4000.
3. Пустая страница — обновить страницу и проверить консоль браузера.

## 6 Завершение работы

1. Выполнить выход из учетной записи.
2. Остановить процессы в терминале (`Ctrl + C`).

# Руководство администратора

## 1 Подготовка окружения

Требуется:

1. Node.js 20+
2. npm
3. Rust (cargo)

## 2 Локальный запуск системы

```bash
cd /Users/akovlevartem/Documents/курсовая
bash start-dev.sh
```

Проверка API:

```bash
curl http://localhost:4000/api/ping
```

## 3 Структура проекта

1. `frontend/` — интерфейс.
2. `backend/` — сервер и БД.
3. `backend/data.db` — рабочая база SQLite.

## 4 Административные задачи

### 4.1 Управление пользователями

1. Создание учетной записи.
2. Назначение роли.
3. Проверка корректности доступа.

### 4.2 Контроль целостности данных

1. Регулярная проверка ключевых таблиц.
2. Мониторинг ошибок в логах backend.
3. Контроль корректности CRUD-операций.

### 4.3 Резервное копирование

1. Остановить сервисы перед копированием.
2. Скопировать `backend/data.db` в папку архива.
3. Хранить версии с датой и временем.

1. Backend не запускается: выполнить `cargo check`.
2. Frontend не запускается: выполнить `npm install` и `npm run dev`.
3. Ошибки API: проверить порт 4000 и маршруты `/api/*`.

## 6 Обновление локальной версии

1. Получить изменения из репозитория.
2. Перезапустить backend и frontend.
3. Проверить ключевые сценарии из тест-кейсов.

# Тест-кейсы

## Формат

- ID — идентификатор теста
- Предусловия — что должно быть перед запуском
- Шаги — последовательность действий
- Ожидаемый результат — критерий успешности

## Набор тест-кейсов

| ID | Предусловия | Шаги | Ожидаемый результат |
|---|---|---|---|
| TC-01 | Backend и Frontend запущены | Открыть `http://localhost:3000` | Открывается главная страница без критических ошибок |
| TC-02 | Есть учетная запись администратора | Ввести корректные данные входа | Успешная авторизация, доступ к административным разделам |
| TC-03 | Есть учетная запись клиента | Войти как клиент | Доступ ограничен пользовательскими разделами |
| TC-04 | Авторизация под администратором | Перейти в «Клиенты», создать клиента | Новая запись появляется в таблице |
| TC-05 | Существующий клиент | Изменить данные клиента | Изменения сохраняются и отображаются в интерфейсе |
| TC-06 | Существующий клиент | Удалить запись клиента | Клиент удален из списка |
| TC-07 | Есть тренер и слот | Создать бронирование | Бронирование появляется в разделе «Бронирования» |
| TC-08 | Есть платежные данные | Создать запись платежа | Платеж отображается в списке с корректным статусом |
| TC-09 | Есть данные отчетов | Открыть раздел «Отчеты» и экспорт | Файл отчета формируется без ошибок |
| TC-10 | Есть токен сессии | Выполнить выход из системы | Сессия сбрасывается, защищенные разделы недоступны |
| TC-11 | Backend запущен | Выполнить `curl /api/ping` | Возвращается статус `running` |
| TC-12 | Неверные данные входа | Попытка авторизации | Отображается сообщение об ошибке без падения интерфейса |

## Критерии приемки

1. Не менее 90% тест-кейсов завершаются успешно.
2. Отсутствуют критические ошибки (невозможность входа, потеря данных, падение backend).
3. Все ключевые пользовательские сценарии выполняются в локальном режиме.

# Функциональные модели IDEF0 / IDEF3 / BPMN

## 1 Контекстная модель IDEF0 (A-0)

### Функция
A-0: «Управление процессами спортивного комплекса».

### ICOM-структура

- Inputs: данные клиентов, заявки на запись, платежные данные.
- Controls: регламенты клуба, роли доступа, расписание, тарифы.
- Outputs: записи в БД, отчеты, подтвержденные бронирования, статусы оплат.
- Mechanisms: frontend (React), backend (Rust/Axum), SQLite, администратор/тренер.

## 2 Декомпозиция IDEF0 (A0)

A1 — Управление пользователями и ролями.
A2 — Управление клиентами.
A3 — Управление расписанием и занятиями.
A4 — Управление бронированиями.
A5 — Управление платежами и отчетностью.

## 3 Процессная модель IDEF3 (сценарий обслуживания клиента)

1. Авторизация сотрудника.
2. Поиск или создание клиента.
3. Выбор занятия.
4. Создание бронирования.
5. Фиксация оплаты.
6. Формирование отчета.

## 4 BPMN-представление (упрощенно)

```mermaid
flowchart LR
    S((Старт)) --> A[Авторизация]
    A --> B{Роль}
    B -->|Администратор| C[Управление данными]
    B -->|Тренер| D[Работа с расписанием и клиентами]
    B -->|Клиент| E[Личный кабинет]
    C --> F[Создание/изменение записи]
    D --> F
    E --> G[Запись на занятие]
    G --> H[Платеж]
    F --> I[Сохранение в БД]
    H --> I
    I --> J[Формирование отчета]
    J --> K((Завершение))
```

## Примечание

Для сдачи в печатном виде рекомендуется финальная отрисовка IDEF0/IDEF3/BPMN в draw.io, Bizagi Modeler или аналогичном инструменте с подписями и нумерацией рисунков.

# ER-модель и схема данных

## 1 Сущности

1. `users`
2. `sessions`
3. `members`
4. `trainers`
5. `classes`
6. `bookings`
7. `payments`
8. `receipts`

## 2 ER-диаграмма (концептуальная)

```mermaid
erDiagram
    USERS ||--o{ SESSIONS : has
    USERS ||--o{ MEMBERS : manages
    TRAINERS ||--o{ CLASSES : leads
    MEMBERS ||--o{ BOOKINGS : creates
    CLASSES ||--o{ BOOKINGS : contains
    MEMBERS ||--o{ PAYMENTS : has
    PAYMENTS ||--o{ RECEIPTS : creates

    USERS {
      int id PK
      string name
      string email
      string role
      string password
    }

    SESSIONS {
      int id PK
      int userId FK
      string token
      string expiresAt
    }

    MEMBERS {
      int id PK
      string name
      string phone
      string email
      string status
    }

    TRAINERS {
      int id PK
      string name
      string specialization
    }

    CLASSES {
      int id PK
      string title
      string date
      string time
      int trainerId FK
    }

    BOOKINGS {
      int id PK
      int memberId FK
      int classId FK
      string status
    }

    PAYMENTS {
      int id PK
      int memberId FK
      decimal amount
      string method
      string status
      string date
    }

    RECEIPTS {
      int id PK
      int paymentId FK
      string number
      string createdAt
    }
```

## 3 Ограничения целостности

1. Первичные ключи задаются во всех таблицах.
2. Внешние ключи обеспечивают согласованность связей.
3. Обязательные поля (NOT NULL) используются для критичных атрибутов.
4. Значения статусов ограничиваются допустимыми бизнес-правилами.

## 4 SQL-часть

Фактические SQL-объекты формируются в backend при инициализации БД (`backend/src/main.rs`) и хранятся в локальном файле `backend/data.db`.

```mermaid
flowchart TD
    A[Старт] --> B[Получение логина и пароля]
```

```mermaid
flowchart TD

    C -->|Нет| D[Сообщить: мест нет] --> B
    C -->|Да| E[POST /api/bookings]
    E --> F{Запись в БД успешна?}
    F -->|Нет| G[Сообщить об ошибке]
    F -->|Да| H[Показать подтверждение]
```

```mermaid
flowchart TD

    D -->|Да| F[Построить таблицу/графики]

    G -->|Да| H[Сформировать файл]

    E --> J[Конец]
    I --> J
```
