# ТЕОРЕТИЧЕСКАЯ ЧАСТЬ КУРСОВОЙ РАБОТЫ

## Тема
Разработка локальной информационной системы управления спортивным комплексом на основе Rust, React и SQLite.

## Введение
Цифровизация услуг в сфере спорта и фитнеса требует внедрения информационных систем, обеспечивающих надежный учет клиентов, расписания, оплат, абонементов и отчетности. Ручное ведение данных приводит к дублированию информации, росту количества ошибок и повышению трудозатрат персонала. Применение специализированной информационной системы позволяет централизовать данные, обеспечить их актуальность и повысить скорость принятия управленческих решений.

Для данного проекта выбрана архитектура локального веб-приложения: пользовательский интерфейс функционирует во фронтенде, а бизнес-логика и операции с базой данных реализуются в бэкенде. Такой подход обеспечивает удобство эксплуатации и масштабируемость, а также снижает зависимость от внешней инфраструктуры.

Объект исследования: процессы учета и управления в спортивном комплексе.

Предмет исследования: методы и технологии проектирования и реализации локальной информационной системы для автоматизации указанных процессов.

Цель работы: теоретически обосновать и реализовать архитектуру информационной системы управления спортивным комплексом, обеспечивающую надежность, производительность и удобство использования.

Для достижения цели решаются задачи:
1. Проанализировать предметную область и функциональные требования.
2. Обосновать выбор архитектуры программной системы.
3. Выполнить сравнительный анализ выбранных технологий (Rust, Axum, React, SQLite, Vite).
4. Определить требования к хранению данных, безопасности и качеству ПО.
5. Сформировать методическую основу для практической реализации проекта.

## 1 Теоретические основы разработки информационной системы

### 1.1 Особенности предметной области спортивного комплекса
Спортивный комплекс как объект автоматизации характеризуется большим количеством взаимосвязанных сущностей: клиентами, тренерами, расписанием, услугами, абонементами, посещениями, финансовыми операциями и уведомлениями. Ключевая проблема в таких системах состоит в необходимости поддержки непротиворечивости данных в условиях параллельной работы сотрудников.

С точки зрения теории информационных систем предметная область относится к классу учетно-аналитических систем с транзакционным характером нагрузки. Основными требованиями являются:
1. Оперативная обработка CRUD-операций по сущностям доменной модели.
2. Поддержка ролевого доступа (администратор, тренер, клиент).
3. Сохранение истории операций и возможность формирования отчетности.
4. Обеспечение целостности данных при конкурентном доступе.

Таким образом, для данной предметной области приоритетны предсказуемость поведения системы, надежность хранения данных и контролируемое изменение бизнес-объектов.

### 1.2 Архитектурный подход к построению системы
В проекте применяется клиент-серверный подход в локальном исполнении. Фронтенд отвечает за визуализацию, валидацию пользовательского ввода и интерактивность интерфейса. Бэкенд реализует маршрутизацию запросов, бизнес-правила, проверку прав доступа, а также взаимодействие с базой данных.

Такое разделение соответствует принципу separation of concerns и обеспечивает:
1. Снижение связанности компонентов системы.
2. Упрощение сопровождения и тестирования.
3. Возможность независимого развития интерфейса и серверной логики.

В бэкенде используются REST-подобные HTTP-эндпоинты и JSON-формат обмена, что является де-факто стандартом интеграции веб-компонентов и позволяет обеспечить прозрачность контрактов между слоями.

### 1.3 Обоснование выбора стека технологий

#### Rust и Axum для серверной части
Язык Rust выбран для бэкенда как технология системного уровня с акцентом на безопасность памяти и предсказуемость исполнения. Ключевая особенность Rust — модель владения (ownership) и заимствования (borrowing), позволяющая предотвращать целый класс ошибок на этапе компиляции, без использования сборщика мусора. Для прикладной серверной разработки это снижает риск критических дефектов и делает эксплуатацию более стабильной.

Фреймворк Axum предоставляет типобезопасную модель обработки запросов через extractors (`State`, `Json`, `Path`, `Query` и др.) и хорошо интегрируется с экосистемой Tokio/Tower. Теоретическое преимущество такого подхода заключается в явном описании входных данных обработчика и сокращении неявных преобразований.

#### React для пользовательского интерфейса
React выбран как компонентная библиотека для построения интерфейсов с декларативной моделью рендеринга. В теории реактивных интерфейсов компонентная декомпозиция улучшает повторное использование кода и управляемость состояния приложения. Принципы иерархии компонентов и однонаправленного потока данных позволяют контролируемо масштабировать UI без роста архитектурной энтропии.

#### SQLite для хранения данных
SQLite является встраиваемой транзакционной СУБД, реализуемой как библиотека, а не отдельный серверный процесс. Для локального проекта это принципиально важно: уменьшается операционная сложность, не требуется отдельное администрирование сервера БД, сохраняется ACID-транзакционность. Концепция serverless-СУБД делает SQLite рациональным выбором для курсового проекта и локальной эксплуатации.

#### Vite как инструмент сборки фронтенда
Vite обеспечивает быстрый старт среды разработки, ESM-ориентированную модель dev-сервера, HMR и оптимизированную production-сборку. С теоретической точки зрения это сокращает цикл обратной связи при разработке интерфейса и повышает производительность инженерной работы.

### 1.4 Принципы проектирования модели данных
Для рассматриваемой системы целесообразна реляционная модель хранения с явной нормализацией ключевых сущностей:
1. `users` — учетные записи и роли.
2. `members` — карточки клиентов.
3. `trainers` — сведения о тренерах.
4. `classes`/`calendar_slots` — расписание и слоты.
5. `bookings` — бронирования.
6. `payments`/`receipts` — финансовые операции.
7. `sessions` — управление аутентификационными сессиями.

Теоретическая цель нормализации — снижение избыточности и предотвращение аномалий обновления. При этом в прикладной реализации допускается денормализация отдельных полей для ускорения чтения отчетных представлений, если это не нарушает целостность.

### 1.5 Требования к качеству и надежности программной системы
Система должна удовлетворять следующим качественным атрибутам:
1. Функциональная корректность: соответствие реализованных сценариев бизнес-требованиям.
2. Надежность: устойчивость к ошибочным данным и отказоустойчивое поведение при сбоях.
3. Производительность: приемлемое время ответа пользовательского интерфейса и API.
4. Сопровождаемость: модульность, читаемость кода, предсказуемая структура проекта.
5. Информационная безопасность: базовая аутентификация, разграничение доступа по ролям, контроль операций изменения данных.

Для бэкенда на Rust качество дополнительно обеспечивается строгой статической типизацией, компиляционными гарантиями модели владения и явной обработкой ошибок.

### 1.6 Информационная безопасность и защита данных
Для локальной учетной системы обязательными являются:
1. Безопасное хранение учетных данных пользователей (хеширование паролей).
2. Контроль сессионных токенов и сроков их действия.
3. Ролевая авторизация для критических операций (например, администрирование базы и пользователей).
4. Валидация входных данных на уровне API и клиентской формы.

Указанные меры не исчерпывают полный набор требований ИБ для промышленной эксплуатации, но формируют необходимый базовый уровень защиты для учебного проекта.

## 2 Нормативная база и требования к оформлению по ГОСТ

### 2.1 Базовый стандарт структуры работы
В качестве основного стандарта структуры и оформления научно-технического отчета применяется ГОСТ 7.32-2017. В стандарте зафиксированы:
1. Состав структурных элементов (введение, основная часть, заключение, список источников, приложения).
2. Общие требования к оформлению текста.
3. Правила построения разделов, подразделов, таблиц, рисунков, формул и приложений.

Согласно ГОСТ 7.32-2017, текст выполняется на листах формата А4, обычно через 1,5 интервала, размер шрифта — не менее 12 pt, рекомендуемый шрифт — Times New Roman; также заданы размеры полей (левое 30 мм, правое 15 мм, верхнее и нижнее по 20 мм) и абзацный отступ 1,25 см.

### 2.2 Оформление списка источников и ссылок
Для библиографического описания источников рекомендуется использовать ГОСТ Р 7.0.100-2018. Стандарт определяет состав и порядок элементов описания и разграничивает его с правилами оформления библиографических ссылок.

Для ссылок на электронные документы, размещенные в сети Интернет, актуален ГОСТ Р 7.0.108-2022. Он уточняет правила указания электронных ресурсов и ориентирован на корректную идентификацию версии документа через URL.

ГОСТ Р 7.0.5-2008 сохраняет значение как стандарт по общим правилам составления библиографических ссылок.

### 2.3 Практическое применение ГОСТ в курсовой работе
Для данной курсовой работы рекомендуется применять следующий минимальный набор:
1. ГОСТ 7.32-2017 — структура и общее оформление текста.
2. ГОСТ Р 7.0.100-2018 — оформление списка использованных источников.
3. ГОСТ Р 7.0.108-2022 — ссылки на интернет-источники.
4. ГОСТ Р 7.0.5-2008 — общие правила библиографических ссылок.

Примечание: кафедральные методические указания могут уточнять требования к титульному листу, нумерации рисунков, объему глав и формату приложений. Эти локальные требования имеют приоритет в рамках конкретного вуза.

## Заключение по теоретической части
Проведенный теоретический анализ показывает, что для задачи автоматизации управления спортивным комплексом целесообразна локальная клиент-серверная информационная система с фронтендом на React и бэкендом на Rust (Axum), использующая SQLite в качестве транзакционной СУБД.

Выбор данного стека обоснован требованиями к надежности, предсказуемости и простоте сопровождения. Rust обеспечивает строгие гарантии корректности на уровне компиляции, Axum — типобезопасную обработку HTTP-запросов, React — управляемую компонентную модель интерфейса, SQLite — простое и надежное локальное хранение данных.

Нормативной основой оформления курсовой работы выступают стандарты серии СИБИД (в первую очередь ГОСТ 7.32-2017, ГОСТ Р 7.0.100-2018 и ГОСТ Р 7.0.108-2022), что позволяет привести текст и библиографический аппарат в соответствие с требованиями российской системы стандартизации.
